**Update: root cause is broader than nvidia-powerd — any GPU register read stalls the machine; memory-clock switches do it in bursts**

Further investigation shows my original attribution to `nvidia-powerd` was incomplete. powerd is just one *client* of a more general problem, and disabling it does not fix the audio glitches (it actually makes them worse — see below). Corrected findings:

**1. Any NVML/GPU register read stalls the entire machine for ~4–5 ms.**
Four RT-priority latency canaries (SCHED_FIFO 70, 1 ms sleep loop, logging >3 ms overshoots) pinned to different cores record *identical* stall timestamps — 117/117 stalls simultaneous across all 24 CPUs, i.e. a whole-machine/bus stall, not scheduler jitter. Polling `nvidia-smi --query-gpu=power.draw` at 2 Hz raises the stall rate by exactly ~2/s (3.65/s → 5.30/s), so the mapping is ~1:1 read→stall.

**2. Memory-clock/pstate switches trigger dense stall storms.**
Under gaming load the driver produces bursts of 10–35 stalls/second (one every ~25 ms) lasting seconds. These storms coincide with mem-clock/pstate transitions and are what actually breaks userspace: PipeWire client streams log xrun bursts and audio audibly glitches exactly during storm windows (verified live — a listener called out glitches in real time against a wall-clock stall log, 4/4 matches). Steady stalls at up to ~5/s are absorbed silently.

**3. A power-limited GPU makes it constant.**
With Dynamic Boost off (`nvidia-powerd` disabled) this 80 W-base-TGP laptop GPU is power-starved under load and flaps mem clock/pstate continuously (P4/9001 ↔ P2/11001 several times a minute) — so glitches become near-constant. This is why disabling powerd worsened the symptom my original report blamed powerd for.

**4. Workaround that resolves the audible symptom:**
```
nvidia-smi -lmc 14001,14001
```
Pinning the memory clock stops the switches; audio xruns drop ~10× immediately and audible glitches stop, at full performance (powerd enabled, P0, mem steady, 150–169 W sustained).

**5. Residual storm mode with mem clock locked.**
With mem pinned, a weaker storm mode remains during *bursty/oscillating* GPU load (utilization swinging 43–99%, SM clock 1485–2512 MHz): one ~3-minute episode logged ~2300 stalls and ~50 xruns while the mem clock never moved. Suspect: power-management activity around the (unlocked) graphics clock. Under steady high load the machine is quiet (~1 stall/s).

**Related issues that look like the same underlying problem:** #1061 (nvidia-powerd microstuttering, RTX 4060 Laptop — matches my original wrong attribution; powerd is a stall *client*, not the cause) and #659 (periodic ~500 ms freezes with GSP RPC timeouts, RTX 4070 Laptop, absent on the proprietary driver). On Blackwell laptops neither escape hatch exists — RTX 50 requires the open modules and GSP — so clock pinning is currently the only mitigation.

**Environment:** Lenovo Legion Pro 7 16IAX10H, RTX 5080 Laptop (GB203M, GSP), driver 610.43.03 (open modules), kernel 7.1.3-2-cachyos (PREEMPT), Core Ultra 9 275HX.

**Tooling/method** (RT canary + stall/xrun/GPU-state monitor): https://github.com/radical89/legion-pro7-linux — `tools/`, README issue #8 has the full writeup. Happy to provide logs or run diagnostics against test builds.
